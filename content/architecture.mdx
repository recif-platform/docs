## Two-Layer Architecture

Recif cleanly separates **governance** from **runtime** into two distinct layers. This is the core architectural insight: how you _build_ your agent should be independent from how you _deploy and manage_ it.

```
┌────────────────────────────────────────────────────────┐
│                    Recif (Governance)                   │
│                                                        │
│  ┌───────────┐  ┌────────┐  ┌───────┐  ┌───────────┐ │
│  │ Dashboard  │  │  API   │  │  CLI  │  │ Operator  │ │
│  │ (Next.js)  │  │  (Go)  │  │(recif)│  │(kubebuildr)│ │
│  └───────────┘  └────────┘  └───────┘  └───────────┘ │
│                                                        │
│  Registry │ Eval │ Deploy │ Governance │ Monitoring    │
└────────────────────────┬───────────────────────────────┘
                         │ gRPC (bidirectional streaming)
                         │ mTLS via Istio
┌────────────────────────┼───────────────────────────────┐
│               Corail Pods (Runtime)                     │
│                        │                                │
│  ┌─────────────────────┼─────────────────────────┐     │
│  │              Agent Pod                         │     │
│  │  ┌──────────┐ ┌──────────┐ ┌───────────────┐ │     │
│  │  │  Corail   │ │  Your    │ │ Istio Sidecar │ │     │
│  │  │  Runtime  │ │  Agent   │ │   (Envoy)     │ │     │
│  │  │          │ │  Code    │ │               │ │     │
│  │  └──────────┘ └──────────┘ └───────────────┘ │     │
│  └───────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────┘
```

### Recif — The Governance Layer

Written in **Go**, Recif handles everything that happens _around_ your agents:

- **Agent Registry**: versioned catalog with metadata, risk profiles, release history
- **Evaluation Engine**: MLflow-integrated pipeline with 14+ scorers, quality gates
- **Deployment Manager**: canary, blue/green, A/B releases via Istio traffic splitting
- **Governance Engine**: risk profiles, guards (PII, secrets, budget), deny-by-default policies
- **Kubernetes Operator**: kubebuilder-based, reconciles Agent CRDs into running pods
- **Dashboard**: Next.js UI for managing agents, viewing evals, monitoring health
- **CLI** (`recif`): developer-facing commands for init, register, eval, deploy, radar

### Corail — The Runtime Layer

Written in **Python**, Corail is the universal adapter that runs inside each agent pod:

- **Framework Agnostic**: plugs into ADK, LangChain, CrewAI, or custom agent strategies
- **Channel Support**: REST API, embeddable widget, Slack, Discord (via channel registry)
- **Tool/MCP Wiring**: connects to any MCP server, wraps CLI tools, manages tool permissions
- **LLM Abstraction**: unified interface for OpenAI, Ollama, Vertex AI, Bedrock, Anthropic
- **Memory Management**: conversation memory, agent memory, knowledge base integration
- **Event Bus**: all actions emit events (message.received, tool.called, guard.blocked) for audit and monitoring

## Kubernetes-Native Design

Every agent in Recif is a Kubernetes resource:

```yaml
apiVersion: recif.dev/v1alpha1
kind: Agent
metadata:
  name: support-agent
  namespace: team-engineering
spec:
  runtime:
    image: ghcr.io/sciences44/corail:latest
    strategy: react
  llm:
    provider: openai
    model: gpt-4o
  governance:
    riskProfile: medium
    guards: [pii-detection, secrets-detection, budget]
    budgetTokensPerDay: 500000
```

The Recif operator watches for Agent CRDs and reconciles them into:

1. A **Deployment** running the Corail image with your agent config
2. A **Service** for internal communication
3. An **Istio VirtualService** for traffic routing and canary releases
4. **ConfigMaps** for agent configuration (SOUL.md, AGENTS.md)
5. **Secrets** for LLM API keys and tool credentials

### Namespace-per-Team Isolation

Multi-tenancy is achieved through Kubernetes namespaces:

```
recif-system/          # Platform components
team-engineering/      # Engineering team's agents
team-support/          # Support team's agents
team-marketing/        # Marketing team's agents
```

Each namespace gets its own:
- Resource quotas (CPU, memory, tokens)
- Network policies (agents can only reach their own tools)
- RBAC roles (team members manage their own agents)
- Budget limits and governance rules

## Communication: gRPC Bidirectional Streaming

Recif and Corail communicate over **gRPC** with bidirectional streaming, secured by Istio mTLS:

**Recif to Corail** (control plane):
- `UpdateConfig` — push new configuration
- `ReloadTools` — hot-reload tool definitions
- `Pause` / `Resume` — traffic control
- `Terminate` — graceful shutdown

**Corail to Recif** (event stream):
- Audit events (every message, tool call, guard decision)
- Monitoring metrics (latency, token usage, error rates)
- Evaluation results
- Governance compliance reports

## Multi-Level Context Hierarchy

Context cascades from platform down to session, with each level inheriting from the one above:

```
PLATFORM (Recif global)
│  Platform-wide settings, default guards, model whitelist
│
├── TEAM (Kubernetes namespace)
│   │  Team tools, knowledge bases, budget, team guards
│   │
│   ├── AGENT (pod)
│   │   │  Agent identity (SOUL.md), operational rules
│   │   │
│   │   ├── USER (per-request, from JWT)
│   │   │   Preferences, permissions, personal memory
│   │   │
│   │   └── SESSION (per-conversation)
│   │       Ephemeral conversation state
```

| Level | What | Set By | Stored In |
|---|---|---|---|
| Platform | Default guards, model whitelist, cost limits | Platform admin | Recif DB / ConfigMap |
| Team | Tools, KBs, team budget, team guards | Team admin | Namespace ConfigMap |
| Agent | Identity (SOUL.md), rules (AGENTS.md) | Agent CRD | Pod volume |
| User | Preferences, permissions, memory | JWT + DB | Recif DB |
| Session | Conversation state | Auto | PostgreSQL |

A key insight: **user preferences propagate across agents**. If a user sets `language=fr`, every agent they talk to knows this — because user context is stored at the platform level, not inside individual agent pods.

## Event-Driven Architecture

An internal EventBus connects all Corail components:

```
message.received → Guard Pipeline → Agent Loop → tool.called
                                                → llm.invoked
                                                → memory.updated
                                  → response.sent → Audit
                                                  → Monitoring
                                                  → Eval (if enabled)
```

Every event is:
- **Typed** — strongly typed event definitions
- **Auditable** — persisted for compliance and debugging
- **Subscribable** — any component can react to any event
- **Traceable** — linked to MLflow traces via OpenTelemetry

## GitOps: Agent Artifacts

Agent configuration is stored as immutable versioned artifacts in a dedicated Git repository (`recif-state`):

```
recif-state/
├── agents/
│   ├── support-agent/
│   │   ├── current.yaml        # Active configuration
│   │   └── releases/
│   │       ├── v1.yaml         # Immutable release artifact
│   │       ├── v2.yaml
│   │       └── v3.yaml
│   └── sre-bot/
│       ├── current.yaml
│       └── releases/
│           └── v1.yaml
├── skills/
├── policies/
└── platform.yaml
```

This enables:
- Full audit trail (every change is a Git commit)
- Diff any two versions
- ArgoCD integration (v0.2) for automated GitOps
- Rollback to any previous version instantly

## Technology Stack

| Component | Technology | Why |
|---|---|---|
| API Server | Go | Performance, strong typing, K8s ecosystem |
| Dashboard | Next.js + React | SSR, modern frontend, Tailwind |
| Operator | kubebuilder (Go) | Native K8s operator pattern |
| Runtime | Python (Corail) | AI/ML ecosystem, framework compatibility |
| Database | PostgreSQL | Reliable, pgvector for embeddings |
| Service Mesh | Istio + Envoy | mTLS, traffic splitting, observability |
| Evaluation | MLflow | Industry standard, GenAI scorers, traces |
| Packaging | Helm | Standard K8s packaging |
| Ingestion | Maree (Python) | Pluggable pipeline, Docling, evaluation-first |
