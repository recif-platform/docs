16:Tfbf,# Canary Deployments

Canary deployments let you test a new agent configuration (model, tools, prompt) on a percentage of live traffic before promoting it to all users.

## How it works

1. **Create a release** — change the agent config (add tools, switch model, update prompt). A new release is saved in the git-state repo.
2. **Start a canary** — select the release and a traffic weight (e.g., 10%). A second pod is created with the new config alongside the stable pod.
3. **Monitor** — compare traces and metrics in MLflow between stable (v1) and canary (v2).
4. **Promote or rollback** — if the canary performs well, promote it to stable. Otherwise, roll back.

```
                    ┌─────────────┐
           90% ───▶ │  Stable v1  │
User ──▶ Proxy ───┤              │
           10% ───▶ │  Canary v2  │
                    └─────────────┘
```

## Channel support

Canary routing depends on how traffic reaches the agent:

| Channel | Canary support | How it works |
|---------|---------------|--------------|
| **REST** (dashboard chat) | Full canary routing | API proxy routes X% of requests to the canary pod |
| **Discord, Slack, WebSocket** | Blue-green via promote | Only one bot connection per token — canary is tested via dashboard, then promoted |

### Why non-HTTP channels use blue-green

Discord, Slack, and WebSocket channels maintain a persistent connection (bot token → platform API). Only one pod can hold the connection at a time. Splitting individual messages between stable and canary would break conversation continuity.

**Recommended workflow for non-HTTP channels:**

1. Start a canary with the new config
2. Test the canary through the **dashboard chat** (REST traffic is split)
3. Evaluate quality in MLflow (traces, scores, latency)
4. **Promote** — the stable pod restarts with the new config, the Discord/Slack bot reconnects automatically

This gives you the same safety net (test before deploy) without breaking the channel's connection model.

## Starting a canary

### From the dashboard

1. Go to **Agents → your agent → Releases**
2. Click **Canary** on the release you want to test
3. Set the traffic weight (1-50%)
4. The canary pod starts automatically

### From the API

```bash
curl -X POST /api/v1/agents/{id}/canary \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"challenger_version": 2, "weight": 10}'
```

### From kubectl

```bash
# Check canary status
kubectl get agent my-agent -n team-default -o jsonpath='{.spec.canary}'
```

## What the canary gets

The canary pod receives the full config from the release artifact:

- Model (provider + ID)
- System prompt / prompt ref
- Tools and skills
- Strategy and channel

The canary always uses the **agent's current image** (`spec.image` from the CRD), not the image snapshot from the release artifact.

## Adjusting traffic weight

```bash
curl -X PATCH /api/v1/agents/{id}/canary \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"weight": 50}'
```

Weight range: 1-99%. The stable pod gets `100 - weight`%.

## Monitoring in MLflow

Both stable and canary report to the same MLflow experiment (`recif/agents/{name}`). They register different LoggedModels:

- Stable: `{name}-v1`
- Canary: `{name}-v2`

Each trace is tagged with `recif.agent_version` so you can filter and compare.

## Promote

Promoting makes the canary config the new stable:

```bash
curl -X POST /api/v1/agents/{id}/canary/promote \
  -H "Authorization: Bearer $TOKEN"
```

This:
1. Deletes the canary pod, service, and Istio resources
2. Updates the CRD `spec.version` to the canary's release version
3. The operator creates a new stable deployment with the new config
4. Non-HTTP channels (Discord, Slack) reconnect automatically

## Rollback

```bash
curl -X POST /api/v1/agents/{id}/canary/rollback \
  -H "Authorization: Bearer $TOKEN"
```

This deletes the canary without changing the stable deployment. Traffic returns to 100% stable.
