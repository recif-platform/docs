16:Tdf6,# Releases

Every configuration change to an agent creates an immutable **release** — a versioned snapshot stored in the git-state repository. Releases are the foundation for safe deployments, canary testing, and rollback.

## Lifecycle

```
Create agent ──▶ Release v1 (auto) ──▶ Deploy v1
                                            │
            Config change ──▶ Release v2    │
                                │           │
                    ┌───────────┤           │
                    ▼           ▼           │
                 Canary      Deploy v2 ◀───┘
                    │
              ┌─────┴─────┐
              ▼            ▼
           Promote      Rollback
```

### 1. Agent creation

When you create an agent (wizard or API), the platform:
1. Creates the Agent CRD in Kubernetes
2. The operator deploys a pod with the initial config
3. A release v1 is automatically created in git-state

### 2. Configuration change

Any config update (model, tools, prompt, strategy) creates a new release:
- The release is saved to the git-state repo (`agents/{namespace}/{slug}/releases/v{n}.yaml`)
- The running agent is **not affected** — the new config is stored but not deployed
- You can review the diff between the current version and the new release in the dashboard

### 3. Deployment

You have three options to deploy a new release:

| Method | How | Risk |
|--------|-----|------|
| **Direct deploy** | Click "Deploy" on a release | Immediate — all traffic switches |
| **Canary** | Click "Canary" with a traffic weight | Gradual — test on a % of traffic first |
| **Rollback** | Deploy an older release | Reverts to a known-good config |

## Release artifact

Each release is a YAML file containing the complete agent configuration:

```yaml
apiVersion: recif.dev/v1
kind: AgentArtifact
metadata:
  name: my-agent
  version: 2
  timestamp: "2026-04-18T10:00:00Z"
agent:
  model:
    provider: vertex-ai
    id: gemini-2.5-flash
  system_prompt: "You are a helpful assistant."
  tools:
    - web_search
    - fetch_url
  skills: []
runtime:
  image: ghcr.io/recif-platform/corail:latest
  channel: rest
  strategy: agent-react
  replicas: 1
```

## Git-state repository

Releases are stored in a dedicated GitHub repository (`recif-state`):

```
agents/
  team-default/
    my-agent/
      current.yaml          # Points to the active release
      releases/
        v1.yaml             # Initial config
        v2.yaml             # After config change
        v3.yaml             # After another change
```

This provides:
- **Audit trail** — every change is a git commit with timestamp and author
- **Immutability** — releases are never modified, only new ones are created
- **Rollback** — deploy any previous release at any time

## Dashboard

Go to **Agents → your agent → Releases** to:
- View all releases with their version numbers
- See the diff between releases
- Deploy or start a canary from any release
- Track which version is currently active

## API

```bash
# List releases
curl /api/v1/agents/{id}/releases \
  -H "Authorization: Bearer $TOKEN"

# Get a specific release
curl /api/v1/agents/{id}/releases/2 \
  -H "Authorization: Bearer $TOKEN"

# Deploy a release (makes it the active version)
curl -X POST /api/v1/agents/{id}/deploy \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"version": 2}'

# View diff between current and latest
curl /api/v1/agents/{id}/releases/diff \
  -H "Authorization: Bearer $TOKEN"
```
