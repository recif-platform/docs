Releases#

Every configuration change to an agent creates an immutable release — a versioned snapshot stored in the git-state repository. Releases are the foundation for safe deployments, canary testing, and rollback.

Lifecycle#

1. Agent creation#

When you create an agent (wizard or API), the platform:

  1. Creates the Agent CRD in Kubernetes
  2. The operator deploys a pod with the initial config
  3. A release v1 is automatically created in git-state

2. Configuration change#

Any config update (model, tools, prompt, strategy) creates a new release:

  • The release is saved to the git-state repo (agents/{namespace}/{slug}/releases/v{n}.yaml)
  • The running agent is not affected — the new config is stored but not deployed
  • You can review the diff between the current version and the new release in the dashboard

3. Deployment#

You have three options to deploy a new release:

MethodHowRisk
Direct deployClick "Deploy" on a releaseImmediate — all traffic switches
CanaryClick "Canary" with a traffic weightGradual — test on a % of traffic first
RollbackDeploy an older releaseReverts to a known-good config

Git-state repository#

Releases are stored in a dedicated GitHub repository (recif-state):

This provides:

  • Audit trail — every change is a git commit with timestamp and author
  • Immutability — releases are never modified, only new ones are created
  • Rollback — deploy any previous release at any time

Dashboard#

Go to Agents → your agent → Releases to:

  • View all releases with their version numbers
  • See the diff between releases
  • Deploy or start a canary from any release
  • Track which version is currently active
